# 008 — Calendar Booking Skill 建置與 OAuth2 授權除錯

> **日期**：2026-03-17  
> **Skill**：`calendar-booking`  
> **憑證類型**：OAuth2 Desktop (installed) App

---

## 1. 問題現象與根因 (Phenomenon & Cause)

本次作業涵蓋三個階段的問題：

### 問題 A — 沙盒環境無法開啟瀏覽器授權

- **問題現象：** 執行腳本時拋出 `webbrowser.Error: could not locate runnable browser`，OAuth2 授權流程中斷。
- **發生原因：** OpenClaw 沙盒環境為 headless Linux，無圖形介面與瀏覽器，`InstalledAppFlow.run_local_server()` 預設嘗試開啟瀏覽器失敗。

### 問題 B — OAuth2 redirect URI 使用 HTTP 被拒絕

- **問題現象：** 手動貼回授權網址後，拋出 `InsecureTransportError: OAuth 2 MUST utilize https`。
- **發生原因：** `oauthlib` 預設強制要求 HTTPS redirect URI，而 `http://localhost:1` 為 HTTP 協定，未設定 `OAUTHLIB_INSECURE_TRANSPORT=1` 環境變數。

### 問題 C — FreeBusy API 回傳 400 Bad Request

- **問題現象：** Token 取得成功，但呼叫 `freebusy().query()` 時回傳 `HttpError 400: Bad Request`。
- **發生原因：** `timeMin` / `timeMax` 參數缺少時區資訊（僅 `2026-03-18T09:00:00`），Google Calendar API 要求 RFC 3339 格式（需含時區偏移，如 `+08:00`）。

### 附帶問題 — Scope 不足導致 403

- **問題現象：** 首次授權使用 `calendar.events.readonly` scope，呼叫 freebusy API 時回傳 `HttpError 403: Insufficient Permission`。
- **發生原因：** `calendar.events.readonly` 不涵蓋 freebusy 端點。需使用 `calendar` 或 `calendar.readonly` 以上層級的 scope。

---

## 2. 解決方案 (Resolution)

- **處理步驟：**
  1. **新增 Console 授權流程**：偵測到無瀏覽器環境時，改用手動模式 — 印出授權 URL，使用者在外部瀏覽器完成授權後，將 redirect URL 貼回 stdin。
  2. **設定環境變數**：執行腳本前加上 `OAUTHLIB_INSECURE_TRANSPORT=1`，允許 HTTP redirect URI。
  3. **統一 Scope**：將 `gcal_freebusy.py` 和 `gcal_create_event.py` 統一使用 `https://www.googleapis.com/auth/calendar`（完整讀寫），共用同一個 `token.json`。
  4. **修正時間格式**：使用 `zoneinfo.ZoneInfo` 將 `timeMin` / `timeMax` 附加時區資訊（如 `2026-03-18T09:00:00+08:00`），符合 RFC 3339。
  5. **刪除舊 Token 重新授權**：因 scope 變更，刪除 `token.json` 後重新走授權流程取得新 token。

- **最終結果：**
  - ✅ FreeBusy 查詢正常回傳可用時段（已排除已佔用時段）
  - ✅ 事件建立成功（回傳 event_id 與 Calendar 連結）
  - ✅ Buffer 機制正確運作（事件間保留 15 分鐘間隔）
  - ✅ Token 快取正常，後續呼叫免重新授權

## 3. 常用指令與操作 (Commands & Operations)

- **查詢可用時段：**

  ```bash
  OAUTHLIB_INSECURE_TRANSPORT=1 python3 scripts/gcal_freebusy.py \
    --credentials "client_secret_xxx.json" \
    --token "token.json" \
    --calendar-id "primary" \
    --date "2026-03-18" \
    --timezone "Asia/Taipei"
  ```

  - **用途：** 查詢指定日期在營業時段內的所有可用預約時段

- **建立日曆事件：**

  ```bash
  OAUTHLIB_INSECURE_TRANSPORT=1 python3 scripts/gcal_create_event.py \
    --credentials "client_secret_xxx.json" \
    --token "token.json" \
    --calendar-id "primary" \
    --title "桑尼工作室 - 王小明 預約" \
    --start "2026-03-18T10:15:00+08:00" \
    --end "2026-03-18T11:15:00+08:00" \
    --timezone "Asia/Taipei"
  ```

  - **用途：** 在 Google Calendar 建立預約事件並回傳事件 ID 與連結

- **刪除 Token 重新授權：**

  ```bash
  rm -f token.json
  ```

  - **用途：** 當 scope 變更或 token 失效時，刪除快取強制重新授權

- **Dry-run 測試（不呼叫 API）：**

  ```bash
  python3 scripts/gcal_freebusy.py --credentials dummy --calendar-id primary --date 2026-03-18 --dry-run
  ```

  - **用途：** 驗證時段計算邏輯，不需要有效憑證

- **Skill 結構驗證：**
  ```bash
  python3 /app/skills/skill-creator/scripts/quick_validate.py calendar-booking
  ```

  - **用途：** 驗證 Skill 目錄結構、YAML frontmatter 與命名規範是否符合 OpenClaw 規格

---

## 4. 結論 (Conclusion)

OAuth2 Desktop App 在 headless 環境中需實作 console-based 授權流程（手動貼回 redirect URL），並注意三個常見陷阱：HTTP redirect 需設定 `OAUTHLIB_INSECURE_TRANSPORT=1`、datetime 參數必須包含時區偏移符合 RFC 3339、以及 Google Calendar 各端點所需的 scope 層級不同（freebusy 需 `calendar.readonly` 以上）。建議未來部署時改用 Service Account 憑證，可完全避開瀏覽器授權流程的限制。
