# OpenClaw Gateway — Token 與 Pairing 認證問題處理報告

## 1. 問題現象與根因 (Phenomenon & Cause)

* **問題現象：** OpenClaw Gateway 以 Docker 容器運行後，透過瀏覽器 Control UI (`http://127.0.0.1:18789/chat`) 連線時，依序出現以下錯誤：
  1. `unauthorized: gateway token missing`
  2. `unauthorized: gateway token mismatch`
  3. `pairing required`
  4. `disconnected (1006): no reason`

* **發生原因：**
  1. **Token Mismatch：** 使用者手動以 `openssl rand -hex 32` 產生 token 寫入 `.env`（`OPENCLAW_GATEWAY_TOKEN`），但 OpenClaw 內部已在 `~/.openclaw/openclaw.json` 中自動產生並使用了不同的 token。環境變數的值與 config 檔的值不一致，導致 mismatch。
  2. **Pairing Required：** Docker 容器以 `bind=lan` 模式運行，Gateway 綁定在 `0.0.0.0:18789`。主機瀏覽器透過 Docker port mapping（`127.0.0.1:18789 → 容器:18789`）連入時，Gateway 看到的來源 IP 為 Docker bridge IP（`172.18.0.1`），非 loopback 地址，因此強制要求裝置配對（device pairing）。
  3. **Loopback 不可用：** 若改為 `bind=loopback`，Gateway 僅綁定容器內的 `127.0.0.1`，主機透過 port mapping 無法到達容器內的 loopback，導致連線完全失敗。

## 2. 解決方案 (Resolution)

### 問題一：Token Mismatch

* **處理步驟：**
  1. 執行 `docker exec openclaw-gateway openclaw dashboard --no-open` 取得 OpenClaw 自動產生的 token 及 Dashboard URL。
  2. 將 `.env` 中的 `OPENCLAW_GATEWAY_TOKEN` 更新為上述指令輸出的 token 值，確保與 `~/.openclaw/openclaw.json` 內 `gateway.auth.token` 一致。
  3. 重啟容器：`docker compose down && docker compose up -d`。
  4. 使用含 token 的 URL 格式存取 Control UI：`http://127.0.0.1:18789/chat#token=<token值>`（注意是 `#token=`，非 `?token=`）。

* **最終結果：** Token mismatch 錯誤排除，但進入下一階段的 pairing required 問題。

### 問題二：Pairing Required

* **處理步驟：**
  1. 編輯主機上的 `~/.openclaw/openclaw.json`（即容器掛載的 config 檔）。
  2. 在 `gateway` 區塊新增以下設定：
     ```json
     "trustedProxies": ["0.0.0.0/0"],
     "controlUi": {
       "allowInsecureAuth": true,
       "dangerouslyDisableDeviceAuth": true
     }
     ```
  3. 重啟容器：`docker compose down && docker compose up -d`。
  4. 確認 Gateway 日誌顯示 `dangerouslyDisableDeviceAuth=true` 已啟用且無 config 錯誤。

* **最終結果：** Gateway 啟動正常，config 設定生效，device pairing 驗證已停用。

### 最終 `.env` 內容

```env
OPENCLAW_IMAGE=ghcr.io/openclaw/openclaw:latest
OPENCLAW_GATEWAY_TOKEN=ed9bd761e0cd0b196678e84caebfb7a031e252a8945ed0ed
OPENCLAW_GATEWAY_BIND=lan
OPENCLAW_ALLOW_INSECURE_PRIVATE_WS=true
```

### 最終 `openclaw.json` 關鍵區塊

```json
{
  "gateway": {
    "mode": "local",
    "bind": "lan",
    "trustedProxies": ["0.0.0.0/0"],
    "controlUi": {
      "allowedOrigins": [
        "http://localhost:18789",
        "http://127.0.0.1:18789"
      ],
      "allowInsecureAuth": true,
      "dangerouslyDisableDeviceAuth": true
    },
    "auth": {
      "mode": "token",
      "token": "<自動產生的token>"
    }
  }
}
```

## 3. 常用指令與操作 (Commands & Operations)

* **指令一：** `docker exec openclaw-gateway openclaw dashboard --no-open`
  * **用途：** 取得 Gateway 自動產生的 token 及帶 token 的 Dashboard URL。

* **指令二：** `docker exec openclaw-gateway openclaw gateway --help`
  * **用途：** 查看 Gateway 所有可用參數（`--auth`、`--bind`、`--token` 等）。

* **指令三：** `docker exec openclaw-gateway sh -c 'cat /home/node/.openclaw/openclaw.json'`
  * **用途：** 查看容器內實際使用的 config 設定檔內容。

* **指令四：** `docker exec openclaw-gateway printenv OPENCLAW_GATEWAY_TOKEN`
  * **用途：** 確認容器內環境變數的 token 值是否與 config 一致。

* **指令五：** `docker compose down && docker compose up -d`
  * **用途：** 重啟容器以套用新的環境變數或 config 設定。

* **指令六：** `docker logs openclaw-gateway`
  * **用途：** 查看 Gateway 啟動日誌，確認是否有 config 錯誤或認證異常。

## 4. 結論 (Conclusion)

在 Docker 環境中部署 OpenClaw Gateway 時，應使用 `openclaw dashboard --no-open` 取得系統自動產生的 token，而非手動產生，以避免 token mismatch。由於 Docker port mapping 導致 Gateway 將主機連線視為外部來源而觸發 device pairing，需在 `openclaw.json` 中設定 `dangerouslyDisableDeviceAuth: true` 來跳過配對驗證；此為開發/本機使用的權宜措施，正式環境應評估 Tailscale Serve 或 reverse proxy 搭配 `trusted-proxy` 認證模式。
