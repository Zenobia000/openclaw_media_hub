# OpenClaw auth-profiles.json 格式錯誤導致 API 金鑰無法識別

## 1. 問題現象與根因 (Phenomenon & Cause)

* **問題現象：** OpenClaw Gateway 啟動後，Agent 發送訊息時回報錯誤：`No API key found for provider "anthropic". Auth store: /home/node/.openclaw/agents/main/agent/auth-profiles.json`。儘管 `auth-profiles.json` 檔案存在且 API 金鑰內容正確，Gateway 仍無法讀取認證資訊。
* **發生原因：** `init-openclaw.ps1` 初始化腳本產生的 `auth-profiles.json` 使用了錯誤的 JSON 結構。腳本寫入的格式為扁平結構 `{ "anthropic": { "apiKey": "..." } }`，但 OpenClaw 內部認證模組（`resolveApiKeyFromCredential`）要求的格式為以 `profiles` 物件包裝的結構，每個 profile 須包含 `type`、`provider` 及對應憑證欄位。

## 2. 解決方案 (Resolution)

* **處理步驟：**
  1. 確認容器內檔案可讀取且內容正確（排除權限、編碼、BOM 等問題）。
  2. 執行 `openclaw models status` 確認 `Providers w/ OAuth/tokens (0)` — 驗證 Gateway 未識別任何認證 profile。
  3. 透過分析 OpenClaw 打包後的原始碼（`auth-profiles-DRjqKE3G.js`），找到 `resolveApiKeyFromCredential` 函式，確認期望的 JSON 結構為 `{ "profiles": { "<profileId>": { "type", "provider", "token/key" } } }`。
  4. 將 `auth-profiles.json` 改寫為正確格式：
     ```json
     {
       "profiles": {
         "anthropic:manual": {
           "type": "token",
           "provider": "anthropic",
           "token": "<API_KEY>"
         }
       }
     }
     ```
  5. 執行 `openclaw secrets reload` 使變更生效。
  6. 再次執行 `openclaw models status` 確認 `Providers w/ OAuth/tokens (1): anthropic` 已正確識別。
* **最終結果：** 問題已排除。Gateway 成功識別 Anthropic 認證 profile，Agent 可正常呼叫 API。

## 3. 常用指令與操作 (Commands & Operations)

* **指令一：** `openclaw models status`
  * **用途：** 檢視當前模型設定及認證狀態，確認各 provider 的 auth profile 是否被正確載入。
* **指令二：** `openclaw secrets reload`
  * **用途：** 重新載入 secret 參照與 auth-profiles.json，無須重啟 Gateway 即可套用認證變更。
* **指令三：** `openclaw models auth paste-token --provider anthropic`
  * **用途：** 透過互動式介面將 token 寫入 auth-profiles.json（需 TTY 環境）。
* **指令四：** `openclaw doctor`
  * **用途：** 執行全面健康檢查，涵蓋狀態完整性、安全性、技能、插件及記憶體搜尋等診斷項目。

## 4. 結論 (Conclusion)

此問題源於初始化腳本 `init-openclaw.ps1` 產生的 `auth-profiles.json` 未遵循 OpenClaw 內部期望的 `profiles` 包裝結構。建議更新腳本範本，使其產出符合 `{ "profiles": { "<provider>:manual": { "type", "provider", "token" } } }` 格式的檔案，避免使用者填入金鑰後仍無法通過認證。
