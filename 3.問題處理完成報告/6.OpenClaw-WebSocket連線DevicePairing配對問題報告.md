# 002 — WebSocket 連線 pairing required 裝置配對問題

## 1. 問題現象與根因 (Phenomenon & Cause)

* **問題現象：** 瀏覽器開啟 `http://127.0.0.1:18789/` 後，WebSocket 連線失敗，錯誤訊息為 `disconnected (1008): pairing required`。
* **發生原因：** Gateway 綁定 `0.0.0.0` 後，主機瀏覽器的連線經 Docker bridge 網路進入容器，來源 IP 為 `172.18.0.1`（非 loopback）。Gateway 的安全機制規定：loopback 連線自動核准，非 loopback 連線需經 device pairing 審批。因此首次從主機連線時被要求配對。

## 2. 解決方案 (Resolution)

* **處理步驟：**
  1. 執行 `openclaw devices list` 確認有一筆 Pending 配對請求，來源 IP 為 `172.18.0.1`。
  2. 取得 Request ID（UUID 格式），執行 `openclaw devices approve <requestId>` 完成配對。
  3. 重新整理瀏覽器頁面，WebSocket 連線成功建立。
  4. 將裝置配對流程寫入 `init-openclaw.ps1` 步驟 7：提示使用者先開啟瀏覽器，腳本自動輪詢 `openclaw devices list` 偵測 pending 請求，偵測到後自動執行 approve。
* **最終結果：** 裝置配對完成，瀏覽器可正常透過 WebSocket 與 Gateway 通訊。配對為一次性操作，同一瀏覽器後續連線無需重複審批。

## 3. 常用指令與操作 (Commands & Operations)

* **指令一：** `docker compose exec openclaw-gateway openclaw devices list`
  * **用途：** 列出所有 Pending 與已配對的裝置，取得配對請求的 Request ID。
* **指令二：** `docker compose exec openclaw-gateway openclaw devices approve <requestId>`
  * **用途：** 核准指定的裝置配對請求，允許該裝置透過 WebSocket 連線 Gateway。
* **指令三：** `docker compose exec openclaw-gateway openclaw devices revoke --device <id> --role <role>`
  * **用途：** 撤銷已配對裝置的存取權限。

## 4. 結論 (Conclusion)

當 Gateway 綁定非 loopback 位址時，所有連線均需經過 device pairing 審批，這是 OpenClaw 的安全設計。已將自動偵測與審批流程整合至初始化腳本，引導使用者在首次設定時完成配對，避免因不熟悉此機制而卡關。
