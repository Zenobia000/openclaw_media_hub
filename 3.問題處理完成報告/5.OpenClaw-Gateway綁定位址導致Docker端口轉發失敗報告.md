# 001 — Gateway 綁定位址導致 Docker 端口轉發失敗

## 1. 問題現象與根因 (Phenomenon & Cause)

* **問題現象：** Docker 容器 `openclaw-gateway` 狀態顯示 healthy，但從 Windows 主機瀏覽器存取 `http://127.0.0.1:18789/` 回應 `ERR_EMPTY_RESPONSE`；`curl` 亦回報 `(52) Empty reply from server`。
* **發生原因：** `.env` 中設定 `OPENCLAW_GATEWAY_BIND=lan`，Gateway 在容器內僅綁定 `ws://127.0.0.1:18789`（容器 loopback）。Docker 的端口映射（`-p 127.0.0.1:18789:18789`）需要容器內服務綁定 `0.0.0.0`，才能將主機流量轉發至容器。容器內部 `curl healthz` 正常，主機端無法到達，確認為綁定位址問題。

## 2. 解決方案 (Resolution)

* **處理步驟：**
  1. 確認容器內 healthz 正常（`docker exec openclaw-gateway curl -fsS http://127.0.0.1:18789/healthz` 回傳 `{"ok":true}`），排除服務本身故障。
  2. 查閱 `openclaw gateway --help`，確認 `--bind` 參數接受 `loopback|lan|tailnet|auto|custom` 五種模式。
  3. 搜尋 OpenClaw 文件，發現 `bind=custom` 搭配 `gateway.customBindHost` 可指定自訂綁定位址。
  4. 修改 `.openclaw/openclaw.json`，新增 `"bind": "custom"` 與 `"customBindHost": "0.0.0.0"`。
  5. 執行 `docker compose down && docker compose up -d` 重啟容器。
  6. 確認日誌顯示 `listening on ws://0.0.0.0:18789`，主機端 `curl healthz` 回傳 200。
  7. 同步更新 `init-openclaw.ps1`，將初始 `openclaw.json` 模板加入上述兩項設定，確保新用戶不再遇到此問題。
* **最終結果：** 主機端可正常存取 `http://127.0.0.1:18789/`，Gateway 回應正常。

## 3. 常用指令與操作 (Commands & Operations)

* **指令一：** `docker exec openclaw-gateway curl -fsS http://127.0.0.1:18789/healthz`
  * **用途：** 從容器內部測試 Gateway 健康狀態，用於隔離問題在容器內或主機端。
* **指令二：** `curl -v http://127.0.0.1:18789/healthz`
  * **用途：** 從主機端測試 Gateway 連通性，`-v` 顯示完整 HTTP 交握過程。
* **指令三：** `docker logs openclaw-gateway`
  * **用途：** 檢視 Gateway 啟動日誌，確認實際綁定位址與監聽狀態。
* **指令四：** `docker compose down && docker compose up -d`
  * **用途：** 重啟 Docker Compose 服務，使設定變更生效。

## 4. 結論 (Conclusion)

在 Docker 環境中，容器內服務必須綁定 `0.0.0.0` 而非 `127.0.0.1`，端口映射才能正常運作。OpenClaw Gateway 需透過 `gateway.bind=custom` 搭配 `gateway.customBindHost=0.0.0.0` 明確指定，已將此設定寫入初始化腳本以防止日後重複發生。
