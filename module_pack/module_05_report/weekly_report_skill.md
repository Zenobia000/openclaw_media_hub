| **版本** | v1.0 |
| **最後更新** | 2026-03-10 |

---

## 功能描述

每週指定時間觸發，自動彙整本週的日報、CRM 數據、任務完成情況，生成一份結構化週報。

核心價值：**把散落在各處的本週數據（日報、CRM、任務清單）自動彙整成一份可執行的週報，省下每週五花 1 小時手動整理的時間。**

---

## 觸發條件

| 觸發方式 | 說明 |
|----------|------|
| **cron 排程** | 每週五 17:00 執行（預設 `0 17 * * 5`） |
| **手動觸發** | 在 OpenClaw 對話中輸入指令手動執行 |

### 手動觸發範例

```
幫我產出這週的週報
```

---

## 資料來源

| 資料來源 | 取得方式 | 資料內容 |
|----------|----------|----------|
| **本週 daily_brief 紀錄** | 讀取 Notion/Sheets 中本週的日報 | 每日重點摘要、趨勢變化 |
| **CRM 數據** | 讀取 CRM 模組的 Notion/Sheets 資料 | 新詢問數、跟進數、成交數、轉換率 |
| **任務清單** | 讀取任務管理工具或 Notion 任務資料庫 | 任務完成率、未完成項目、blockers |

## 輸入參數

| 參數名稱 | 型別 | 必填 | 預設值 | 說明 |
|----------|------|------|--------|------|
| `week_start` | string | N | 本週一日期 | 週報起始日，格式 `YYYY-MM-DD` |
| `week_end` | string | N | 本週五日期 | 週報結束日，格式 `YYYY-MM-DD` |
| `data_sources` | string[] | N | `["daily_brief", "crm", "tasks"]` | 資料來源清單 |
| `output_destination` | string | N | `"notion"` | 輸出目的地 |
| `recipients` | string[] | N | `["self"]` | 發送對象 |
| `include_metrics` | boolean | N | `true` | 是否包含業務數據 |
| `include_risks` | boolean | N | `true` | 是否包含風險提醒 |

## 執行流程

```
排程到點 / 手動觸發
       │
       ▼
[1] 計算本週日期範圍（週一 ~ 週五）
       │
       ▼
[2] 讀取本週所有 daily_brief 紀錄
       │
       ▼
[3] 讀取 CRM 數據（新 lead、跟進、成交）
       │
       ▼
[4] 讀取任務清單（完成數 / 總數）
       │
       ▼
[5] 彙整 + 生成重點摘要（從日報中提煉本週 Top 5）
       │
       ▼
[6] 計算業務數據 & 與上週比較
       │
       ▼
[7] 生成下週待辦（從未完成任務 + 風險項目推導）
       │
       ▼
[8] 套用固定格式模板輸出
       │
       ▼
[9] 寫入輸出目的地 + 發送通知
       │
       ▼
[10] 寫入執行紀錄
```

## 安全限制

| 規則 | 說明 |
|------|------|
| **只彙整已有數據** | 不編造數字，CRM 數據直接從資料來源讀取 |
| **數據不足時標注** | 如果某資料來源無數據，該區塊標注「本週無數據」而非留空 |
| **不做預測** | 週報只報告事實，不做業務預測（除非使用者明確要求） |
| **敏感數據不外傳** | 業務數據不透過公開管道傳送，Telegram 通知可選擇只發摘要 |

---

## 數據彙整規則

### 重點摘要生成邏輯

1. 從本週 5 份日報中，提取每份日報的 Top 1 重點
2. 合併重複主題（同一事件多天報導的合併為一條）
3. 按重要性排序，保留 3-5 條
4. 每條重點不超過 50 字

### 業務數據計算

| 指標 | 計算方式 |
|------|----------|
| 新 Lead 數 | CRM 中本週新增的 lead 筆數 |
| 跟進數 | CRM 中本週有跟進紀錄的 lead 筆數 |
| 成交數 | CRM 中本週狀態轉為「成交」的筆數 |
| 任務完成率 | (本週完成任務數 / 本週總任務數) * 100% |
| 週環比變化 | 本週數據 - 上週數據 |

### 風險識別規則

自動標記以下情境為風險：

- 任務完成率低於 60%
- 新 Lead 數較上週下降超過 30%
- 超過 3 天未跟進的 lead
- 截止日在下週的未完成任務

## 設定範例

```yaml
weekly_report:
  schedule: "0 17 * * 5"
  timezone: "Asia/Taipei"
  data_sources:
    - daily_brief
    - crm
    - tasks
  output_destination: notion
  recipients:
    - telegram: "@your_username"
    - email: "boss@company.com"
  include_metrics: true
  include_risks: true
  compare_with_last_week: true
  language: "zh-TW"
```

## 錯誤處理

| 錯誤情境 | 處理方式 |
|----------|----------|
| 本週無日報紀錄 | 重點摘要區塊標注「本週無日報紀錄」 |
| CRM 資料讀取失敗 | 業務數據區塊標注「資料讀取失敗」+ 通知管理員 |
| 部分資料來源無數據 | 有數據的正常輸出，無數據的標注說明 |
| Notion API 失敗 | 重試 1 次，仍失敗則輸出到本地檔案 |

## 與其他模組的關係

```
Module 03 (CRM) ──────┐
                       │
Module 05 (daily_brief)┼──→ weekly_report ──→ Module 06 (notification)
                       │
任務清單 ─────────────┘
```

- **上游**：依賴 daily_brief 的每日輸出、CRM 模組的數據
- **下游**：產出的週報透過 Module 06 的通知機制發送

---

## 教學重點

1. **先有日報再有週報** — weekly_report 依賴 daily_brief，先確保日報正常運作
2. **數據來源可漸進式接入** — 先只接日報，再接 CRM，再接任務清單
3. **格式固定是關鍵** — 和 daily_brief 一樣，用 schema + hooks 鎖格式
4. **與上週比較是亮點** — 週環比數據能讓週報真正有價值
