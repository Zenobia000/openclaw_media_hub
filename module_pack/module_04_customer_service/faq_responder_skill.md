# OpenClaw Skill 規格：faq_responder

## 基本資訊

| 欄位 | 值 |
|------|-----|
| Skill 名稱 | `faq_responder` |
| 版本 | 1.0 |
| 模組歸屬 | module_04_customer_service |
| 適用情境 | 客服自動回覆、常見問題處理、客戶訊息分流 |

---

## 功能描述

接收客戶提問，比對 FAQ 知識庫，自動回覆常見問題。超出知識庫範圍或涉及報價、合約、金額、時程承諾的問題，自動升級轉人工處理。

**核心價值**：把 80% 的重複性問題交給 AI，讓人力專注在 20% 真正需要判斷力的事情上。

---

## 觸發條件

| 條件 | 說明 |
|------|------|
| 預設處理器 | 所有透過 Telegram Bot DM 進來的客戶訊息 |
| 優先級 | 最低（其他 Skill 未匹配時才進入 FAQ 處理） |

**處理流程**：
```
客戶訊息進入
  ↓
其他 Skill 是否匹配？（如 lead_capture 的詢問意圖）
  ├─ 是 → 交給對應 Skill
  └─ 否 → 進入 faq_responder
            ↓
        知識庫是否有匹配？
          ├─ 是 → 自動回覆
          └─ 否 → 是否涉及敏感議題？
                    ├─ 是 → 升級轉人工
                    └─ 否 → 回覆「讓我幫您轉接專人」
```

## 知識庫格式

使用 Markdown FAQ 文件作為知識庫，Q&A 對照表格式：

### 檔案結構

```
knowledge_base/
├── faq_general.md        # 一般常見問題
├── faq_service.md        # 服務相關問題
├── faq_process.md        # 流程相關問題
└── faq_pricing.md        # 價目表（僅限有明確公開價格時使用）
```

### FAQ 文件格式範例

```markdown
# 一般常見問題

## Q: 你們的營業時間是什麼？
A: 我們的服務時間為週一至週五 09:00-18:00（台灣時間 UTC+8）。
週末和國定假日休息，但 AI 客服 24 小時在線，人工客服將於上班時間回覆。

## Q: 你們的辦公地點在哪裡？
A: 我們位於台北市信義區 XX 路 XX 號 X 樓。
如需來訪，建議事先預約以確保有人接待。

## Q: 可以提供哪些服務？
A: 我們提供以下服務：
- AI 自動化顧問
- 資料科學培訓課程
- 企業內訓工作坊
- 一對一技術諮詢
詳細內容歡迎進一步詢問。

## Q: 如何開始合作？
A: 合作流程如下：
1. 初步溝通了解需求
2. 提供客製化提案與報價
3. 確認合作內容與時程
4. 簽約執行
第一步只需在這裡告訴我您的需求即可。
```

### 知識庫設計原則

- 每個 Q&A 對獨立成段，方便 AI 精準匹配
- 答案簡潔但完整，不超過 200 字
- 避免模棱兩可的措辭
- 價格相關只放公開價目（如有），不放議價空間
- 定期更新（建議每月審視一次）

## 輸出

### 自動回覆

| 項目 | 說明 |
|------|------|
| 回覆內容 | 基於知識庫的匹配答案，語氣友善專業 |
| 回覆時間 | < 5 秒 |
| 補充動作 | 回覆後詢問「還有其他問題嗎？」 |
| CRM 記錄 | 自動將對話摘要寫入客戶的 AI 互動記錄 |

### 升級通知

| 項目 | 說明 |
|------|------|
| 客戶回覆 | 「讓我幫您轉接專人，請稍候」 |
| 通知內容 | 對話摘要 + 客戶資訊 + 建議優先級 |
| 通知對象 | 指定負責人（Telegram 私訊或 Email） |
| CRM 記錄 | 狀態標為「需人工處理」 |

## 安全限制

### 絕對遵守的規則

| 規則 | 說明 | 違反時的處理 |
|------|------|------------|
| 只回答知識庫內有的內容 | 不根據「一般常識」回答業務相關問題 | 轉人工 |
| 不編造答案 | 知識庫無匹配時，回覆「讓我幫您轉接專人」 | 轉人工 |
| 不報價格 | 除非知識庫有明確公開價目表 | 轉人工 |
| 不承諾交期 | 任何時程相關承諾都需人工確認 | 轉人工 |
| 不處理客訴 | 客戶表達不滿或投訴 | 立即轉人工 |
| 不處理退款 | 任何退款相關請求 | 立即轉人工 |
| 不提供法律建議 | 合約、法律、責任相關問題 | 轉人工 |
| 所有對話回寫 CRM | 確保每次互動都有記錄 | — |

### AI 回覆的語氣規範

```
DO:
- 友善、專業、簡潔
- 使用「您」而非「你」
- 主動提供下一步建議
- 承認不確定時誠實說「我不確定，幫您轉接專人」

DON'T:
- 不使用過度熱情的語氣（不要「太棒了！」「恭喜您！」）
- 不使用 emoji（除非品牌風格明確要求）
- 不用「親」「寶」等過於隨便的稱呼
- 不模仿客戶的非正式語氣
```

## 第一版範圍

### 做的

| 功能 | 說明 |
|------|------|
| 單一 Telegram 入口 | 一個 Bot，DM 模式 |
| FAQ 自動回覆 | 基於 Markdown 知識庫 |
| 升級轉人工 | 超出範圍自動轉接 |
| CRM 記錄 | 對話摘要寫入 CRM |
| 非工作時間回覆 | 自動告知人工將於上班時間回覆 |

### 不做的

| 功能 | 原因 |
|------|------|
| 多群組廣播 | 複雜度高，第一版不需要 |
| Telegram topic routing | OpenClaw 的 topic 功能仍在調整中 |
| 多平台同時上線 | 先做好一個再擴展 |
| 語音/圖片處理 | 第一版只處理文字訊息 |
| 多語言支援 | 先以繁體中文為主 |

## Skill 定義片段（OpenClaw 格式參考）

```yaml
skill:
  name: faq_responder
  description: >
    自動回覆常見問題，超出範圍轉人工
  triggers:
    - type: message
      channel: telegram_dm
      priority: lowest  # fallback handler
  knowledge_base:
    format: markdown
    path: ./knowledge_base/
    files:
      - faq_general.md
      - faq_service.md
      - faq_process.md
      - faq_pricing.md  # optional
  actions:
    - match_faq
    - auto_reply
    - escalate_if_needed
    - log_to_crm
  safety:
    no_hallucination: true
    no_pricing: true  # unless in knowledge base
    no_commitment: true
    escalate_on:
      - pricing_inquiry
      - contract_discussion
      - complaint
      - refund_request
      - legal_question
      - technical_failure
  dependencies:
    - module_03_crm
    - telegram_channel
```

## 課堂提示

- 知識庫的品質決定了客服品質，花時間寫好 FAQ 比調 prompt 更有效
- 第一版的知識庫不需要很大，10-20 組 Q&A 就夠起步
- 上線後第一週，每天看一次 AI 的回覆記錄，找出「回答不好的」和「該轉人工沒轉的」，持續優化
- 客服不是越自動越好，重點是「該自動的自動、該人工的人工」
